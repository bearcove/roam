To design a robust semantic model for "transparent retry" in a bidirectional RPC system, we must first establish a fundamental distinction: the difference between a **Transport Message** (the bytes sent) and a **Logical Operation** (the intended effect).

The core of the problem is that in a distributed system, the "State of the World" is often desynchronized between the Client, the Network, and the Server. A "correct" model must provide a stable anchor for the identity of an operation that persists across these failures.

---

## 1. The Foundation: The Operation Identity ($OpID$)

For a system to reason about "the same logical operation," it must be able to name it. I propose that every RPC must be bound to a **universally unique Operation Identity ($OpID$)** generated by the client.

This $OpID$ is not a sequence number (which is transport-specific); it is a claim of intent. Without an $OpID$, the server cannot distinguish between a "retry of a lost request" and a "new request that happens to look the same."

### The State Machine of a Logical Operation

On the server, an $OpID$ transitions through these conceptual states:

1. **Unknown:** The server has no record of this $OpID$.
2. **In-Flight:** The handler is currently executing.
3. **Terminal (Success/Failure):** The handler finished, and the result is cached.
4. **Tombstoned:** The record of the operation is deleted (after a TTL).

---

## 2. Handling Duplicate Attempts

When a client retries an RPC with the same $OpID$, the server’s behavior depends entirely on the current state of that $OpID$:

* **If In-Flight:** The server must **block or reject** the new request. It should not start a second handler. The second attempt is essentially a "join" request to the existing execution.
* **If Terminal:** The server must **replay** the cached result (Success or Failure) without re-invoking the handler.
* **If Unknown:** This is treated as the first attempt.

### The "Seal" Principle

To ensure correctness, once an operation reaches a **Terminal** state, it is **sealed**. Whether it failed with a "Database Connection Error" or succeeded with "User Created," that is the truth for that $OpID$. If the client retries, it receives that same error. This prevents "partial success" where a retry silently succeeds after a previous attempt left a side effect but returned an error.

---

## 3. The Contract: Three Semantic Categories

A single model is insufficient because the cost of "sealing" and "replaying" varies. I propose three classes of methods that a service developer must explicitly choose from:

| Category | Semantic Guarantee | Server Requirement |
| --- | --- | --- |
| **Idempotent (Natural)** | **At-Least-Once.** Rerunning is always safe (e.g., Read, Put-Overwrite). | No $OpID$ tracking required; just rerun. |
| **Transactional (Replay)** | **Exactly-Once-Effect.** The system ensures the handler runs to completion once. | Must track $OpID$ and cache the result. |
| **Non-Retryable** | **At-Most-Once.** If the first attempt is uncertain, it is a hard failure. | Reject any retry with the same $OpID$. |

---

## 4. Addressing Partial Failures and Cancellation

This is where "simplicity theater" usually fails. If a handler is mid-execution and the client disconnects, what happens?

### The "Ghost" Execution Problem

If the client disconnects, the server-side handler may still be running.

* **Automatic Cancellation:** The RPC runtime should propagate a cancellation signal to the handler.
* **Retry Interaction:** If the client retries while the "ghost" handler is still running, the retry hits the **In-Flight** state. The runtime should wait for the original handler to finish (or fail) and then return that result to the new caller.

### Partial Side Effects

If a handler fails partway through (e.g., crashes or returns an application error) and has already performed side effects, the $OpID$ remains **Terminal (Failure)**.

* **The Runtime's Obligation:** It must guarantee that the $OpID$ state is updated atomically with the handler's completion.
* **The Handler's Obligation:** If the developer chooses "Transactional," they must ensure their side effects are atomic (e.g., inside a DB transaction). If they cannot, they must use "Non-Retryable."

---

## 5. Transport vs. Application Layers

We must draw a hard line between **Session Reliability** and **Operation Reliability**.

1. **Transport/Session Layer (TCP/QUIC/Virtual Conn):** Handles packet loss and temporary reconnection. If the transport can mask a blip without the RPC layer knowing, it should. This does *not* require an $OpID$.
2. **RPC Layer (The $OpID$ Layer):** Handles the case where the Session was lost entirely. The client’s "Retry Logic" lives here. It uses the $OpID$ to ask the server: *"I lost our last conversation; did you finish Task X?"*

---

## 6. Critical Edge Cases & Hidden Failure Modes

* **The Replay Window (TTL):** An $OpID$ cannot live forever. The server must "Tombstone" them after a duration (e.g., 24 hours). If a client retries after the TTL, the server sees it as **Unknown** and may rerun it. This is a **Correctness Violation** if the client is very late.
* *Solution:* The server should reject $OpID$s with a "Creation Timestamp" older than the TTL.


* **The Response-Loss Race:** The server finishes work and crashes *after* committing the result to the $OpID$ cache but *before* sending the response.
* *Result:* On recovery, the client retries, the server sees the **Terminal** state, and replays. This is correct.


* **Cancellation Race:** The client cancels, but the server completes the work just as the cancellation arrives.
* *Result:* The $OpID$ state must be "First One Wins." If the handler finishes, the result is Success. If the cancellation is processed first, the state is Terminal (Cancelled). Any retry gets the winning state.



## Summary of the Proposed Model

1. **Client-Side:** Mandatory $OpID$ and "Request Timestamp" for every logical call.
2. **Server-Side:** An **Atomic Result Store** that maps $OpID \rightarrow \{Status, Result\}$.
3. **Strict Sealing:** Once an $OpID$ is marked Success, Failure, or Cancelled, it never runs again.
4. **Wait-on-Duplicate:** If a retry hits an In-Flight $OpID$, it transparently awaits the original result.

This model moves the burden of "correctness" from the transport to a persistent record of intent. It forces service implementers to decide if they can support the "Transactional" contract or if they must fallback to "Non-Retryable."
